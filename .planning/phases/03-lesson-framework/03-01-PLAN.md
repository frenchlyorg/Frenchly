<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| student browser → Server Component / Server Action | Authenticated student requests cross into server-side DB access; user_id is re-derived server-side, never trusted from client |
| authenticated role → Postgres (RLS) | Students hold the `authenticated` Postgres role; RLS + column grants are the enforcement layer |
| migration/seed → live DB | One-time DDL + seed push mutates production schema |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | profiles.current_level_id column | mitigate | Do NOT grant `update (current_level_id)` to authenticated; Phase 2 grant stays `update (username, updated_at)`. Only service_role sets current_level_id. Asserted by rls-phase3 static test. |
| T-03-02 | Tampering | sub_component_progress writes | mitigate | RLS INSERT/UPDATE `with check ((select auth.uid()) = user_id)` — DB rejects rows for a different user_id. Server Action (Plan 03) re-derives user_id from getUser(). |
| T-03-03 | Information Disclosure | levels/lessons/sub_components content tables | mitigate | Content SELECT policies use `to authenticated` only (never `to anon`); anonymous callers see zero content rows. |
| T-03-04 | Tampering | orphaned progress rows | mitigate | FK `references public.sub_components on delete cascade` and `references auth.users on delete cascade` — progress rows are removed when parent is deleted. |
| T-03-05 | Information Disclosure | sub_component_progress cross-student read | mitigate | SELECT policy `using ((select auth.uid()) = user_id)` scopes reads to the owner. Mirrors Phase 2 SEC-02. |
</threat_model>

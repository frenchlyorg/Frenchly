<objective>
Placement diagnostic vertical slice (DIAG-01) — the thin end-to-end path: a first-time student starts the placement diagnostic, answers 10 server-drawn questions one at a time, the server grades each answer + computes the final score + places the student (admin-client unlock), and a result screen shows the level with no percentage. After this slice, a real user can be placed at French 1 or French 2.

Purpose: This is the core unlock mechanism for first-time students (success criteria 1–3). The security boundary is the whole point: the server draws questions, grades them, computes the score, and writes the watermark via the admin client — the client is never trusted with the answer key or the score.
Output: `src/actions/diagnostic.ts`, its security tests, the placement page flow, and the shared diagnostic UI components (progress, question card, MC button, fill-in input, result).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → submitDiagnosticAnswer (Server Action) | Untrusted answer string + attempt/question ids cross here; validated with zod, graded server-side. |
| client → startPlacementDiagnostic (Server Action) | Start request crosses here; user derived server-side, retake blocked server-side. |
| Server Action → profiles (unlock write) | Privilege boundary — only service_role may advance the watermark. |
| Server Component → diagnostic_questions | Answer key must not cross into the RSC payload. |

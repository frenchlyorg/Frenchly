export function normalizeFillin(raw: string): string {
  if (raw.trim() === '') return ''
  return raw
    .trim()
    .toLowerCase()
    .normalize('NFD')
    .replace(/[̀-ͯ]/g, '')
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB content → server parse | sub_components.content (text) is authored data, not user input. Zod parse provides integrity check at application layer. |
| Client grading → server completion | gradeFillin/gradeMatching run client-side; only sub-component ID (not score) is sent to the server via markSubComponentComplete. Students cannot inflate scores — the server only records completion, not correctness. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | Client-side grading results | accept | Only the sub-component ID is sent to the server (markSubComponentComplete). The server never receives or stores a "score" — completion is binary. RLS ensures students only complete their own rows. Educational integrity is the learning goal, not score integrity. |
| T-05-02 | Denial of Service | parseProblemContent with adversarial JSON | mitigate | Wrapped in try/catch; Zod parse rejects malformed data; returns null → empty-problem fallback renders without crash. |
| T-05-03 | Information Disclosure | correctAnswer in ProblemData prop | accept | Practice problem answers are non-sensitive educational content (not PII, not financial). Answers in client bundle is expected behavior — problems grade client-side by design. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed this phase (Package Legitimacy Audit: none needed). |
</threat_model>

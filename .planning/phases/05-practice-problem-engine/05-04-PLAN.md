<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase DB → server parse | content column contains JSON authored at seed time. parseProblemContent validates at parse; malformed returns null. |
| Server Component → Client Component | problemData (typed ProblemData or null) flows from server to client via props. No correctAnswer exposure risk beyond expected client-bundle behavior (answers are non-sensitive educational content). |
| Client → markSubComponentComplete | Only sub-component ID crosses the client→server boundary. Score is never transmitted. RLS ensures students only write their own rows. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-08 | Tampering | Sub-component ID passed to markSubComponentComplete | mitigate | Server Action already validates: (1) UUID format via Zod, (2) sub-component existence in DB, (3) user_id from server-side session (never from client). Existing implementation from Phase 3. |
| T-05-09 | Denial of Service | Malformed practice content JSON causing lesson page crash | mitigate | parseProblemContent wraps in try/catch; null return → empty-problem fallback renders. Lesson page never throws from a bad practice row. |
| T-05-10 | Information Disclosure | correctAnswer field in client-rendered props | accept | Practice answers are non-sensitive educational content. Client-side grading requires the answer in the component. No PII, no financial data. This is expected behavior. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed (Package Legitimacy Audit: none). |
</threat_model>

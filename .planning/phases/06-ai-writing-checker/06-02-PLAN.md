Request validation schema (Zod):
  - `const RequestSchema = z.object({ subComponentId: z.string().uuid(), text: z.string().min(1).max(4000) })`

POST handler logic — exact sequence:
  1. Auth: `const supabase = await createClient()` then `getUser()` — return 401 if no user
  2. Body parse: `request.json()` in try/catch (return 400 on parse failure), then `RequestSchema.safeParse()` (return 400 on schema failure)
  3. Rate limit: midnight UTC window — `const today = new Date(); today.setUTCHours(0, 0, 0, 0)` — COUNT rows WHERE user_id = user.id AND created_at >= today.toISOString() — if count >= 10, INSERT a row with feedback_text = null (rate-limited row for audit integrity), then return 200 with JSON `{ feedback: "You've used all your writing checks for today — come back tomorrow!", rateLimited: true }` (not 429 — the client handles the message; lessons must never block per D-07)
  4. Anthropic call: wrap in try/catch — on success set feedbackText to response.content[0].text — on catch log error and leave feedbackText = null
  5. DB insert: `supabase.from('writing_submissions').insert({ user_id: user.id, sub_component_id: subComponentId, feedback_text: feedbackText, created_at: new Date().toISOString() })` — use `.upsert(..., { onConflict: 'user_id,sub_component_id', ignoreDuplicates: true })` instead of insert to handle D-12 one-shot gracefully on retry (unique index prevents duplicates; `ignoreDuplicates: true` means no error thrown)
  6. Return: `NextResponse.json({ feedback: feedbackText ?? "We couldn't check that right now — keep going!", rateLimited: false })`

Security constraints (CLAUDE.md):
  - `user_id` is ALWAYS derived from `supabase.auth.getUser()` — never from request body
  - `process.env.ANTHROPIC_API_KEY` is server-only — never pass it to any response
  - Input text is length-capped at 4000 chars by Zod schema — no additional sanitization needed for AI call (student text goes into user message, not system prompt)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (WrittenCard fetch) → /api/check-writing | POST body from untrusted browser; subComponentId and text must be validated |
| /api/check-writing → Anthropic API | Student text forwarded as user message; system prompt is static and controlled |
| /api/check-writing → Supabase DB | rate-limit read + insert; user_id always server-derived |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Information Disclosure | ANTHROPIC_API_KEY in route.ts | mitigate | Key accessed via process.env only; route lives in src/app/api/ (server-only per App Router); never returned in response |
| T-06-05 | Tampering | user_id accepted from request body | mitigate | user_id always from supabase.auth.getUser() server-side; request body schema has no user_id field |
| T-06-06 | Denial of Service | High-volume AI submissions per user | mitigate | Rate limit: 10/day/user enforced before Anthropic call; midnight UTC reset (D-10, D-11) |
| T-06-07 | Tampering | Prompt injection in student text | accept | System prompt is static module-level const; student text is user-turn content only; Haiku processes it as user input, not instructions; risk acceptable at this scale |
| T-06-08 | Tampering | Phantom rate-limit bypass (submitting row already exists) | mitigate | upsert with ignoreDuplicates on (user_id, sub_component_id) unique index; duplicate submissions are silently ignored |
| T-06-09 | Denial of Service | Oversized input text inflating Anthropic cost | mitigate | text.max(4000) in Zod schema caps input tokens; max_tokens: 80 caps output tokens |
| T-06-SC | Tampering | npm install @anthropic-ai/sdk | mitigate | Package legitimacy checkpoint in Task 1; RESEARCH.md confirms official Anthropic maintainers; human verification required before install proceeds |

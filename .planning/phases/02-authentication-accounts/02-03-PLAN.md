<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> deleteAccount action | Deletion must act on the authenticated user only, never a client-supplied id |
| browser -> resetPassword action | Reset must not disclose whether an email is registered |
| reset email link -> callback route | Untrusted code param exchanged for a session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-15 | Elevation of Privilege | Deleting another user's account via forged id | mitigate | deleteAccount resolves the id from getUser server-side; ignores any client-supplied id |
| T-02-16 | Information Disclosure | Email enumeration via password-reset response | mitigate | resetPassword always returns the same vague success message regardless of email existence |
| T-02-17 | Spoofing | Soft-deleted account still able to log in | mitigate | admin.deleteUser(id, true) marks auth.users deleted_at so Supabase rejects login; PII anonymized in profiles (Pitfall 6) |
| T-02-18 | Tampering | Reset callback accepting an invalid/forged code | mitigate | exchangeCodeForSession validates the code server-side; invalid codes redirect to /login?error=invalid-reset-link |
| T-02-19 | Information Disclosure | Service-role key used for deletion leaking to client | mitigate | deleteAccount runs only in the server action via createAdminClient; admin client never imported into a "use client" file |
</threat_model>

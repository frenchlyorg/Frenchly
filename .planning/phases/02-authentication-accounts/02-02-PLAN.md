<objective>
Deliver the core auth loop as one vertical slice: a visitor signs up (email + username + password), the trigger creates their profile, they land on /dashboard greeted by username, the nav reflects auth state, and a returning user logs in with a persistent session. Includes the rate-limit lockout (SEC-04) and vague login errors (SEC-01/D-13).

Purpose: This is the heart of Phase 2 — after this slice a real user can create an account and log in. Everything else (logout, deletion, reset, admin) extends from these actions and forms.

Output: signUp + signIn Server Actions, SignupForm + LoginForm client components, /signup + /login pages, the auth route-group layout, a protected /dashboard stub, and nav/layout wired to live auth state.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form -> signUp/signIn Server Action | Untrusted email/username/password crosses into server validation |
| signIn -> login_attempts table | Rate-limit state shared across serverless instances via Postgres |
| layout/dashboard -> profiles | Server-side reads gated by RLS; getUser verifies identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Elevation of Privilege | Brute-force credential guessing on signIn | mitigate | checkRateLimit counts login_attempts; 6th attempt in 15 min locked out before auth call (SEC-04, D-19) |
| T-02-07 | Information Disclosure | Account enumeration via login/lockout errors | mitigate | Vague "Email or password incorrect." (D-13); lockout applied regardless of whether email exists (Pitfall 7) |
| T-02-08 | Tampering | Malformed/profane username reaching the DB trigger | mitigate | Full server-side username validation (regex + profanity) before auth.signUp; unique-constraint 23505 handled gracefully (Pitfalls 4,5) |
| T-02-09 | Elevation of Privilege | Client tampering with role to reach /admin | mitigate | Role read server-side from profiles in signIn; never from user_metadata; redirect decided server-side (D-16) |
| T-02-10 | Spoofing | Unverified session used for dashboard/nav identity | mitigate | layout + dashboard use getUser() (server-verified), not getSession() |
</threat_model>
